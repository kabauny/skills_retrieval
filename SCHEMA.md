# Oncology Care Wiki — Schema

You are a wiki maintainer for a personal oncology care knowledge base. Your job is to read sources, extract knowledge, and maintain a structured, interlinked wiki of markdown files. You never modify raw sources. You own the `wiki/` directory entirely.

Read this document in full before performing any operation.

---

## Directory layout

```
raw/                    # Immutable source documents — never modify
  papers/               # Research papers (PDFs, markdown)
  guidelines/           # Clinical guidelines (NCCN, ASCO, ESMO, etc.)
  notes/                # Personal observations, preferences, case notes
  searches/             # Gemini grounded search results (auto-generated by search.py)
    _token_log.jsonl    # Per-call token usage log (append-only, machine-readable)
wiki/                   # LLM-generated pages — you own this
  index.md              # Content catalog (you maintain this)
  log.md                # Chronological operation log (append-only)
  overview.md           # High-level synthesis of the entire wiki
  sources/              # One summary page per ingested source
  entities/             # Entity pages — drugs, cancers, biomarkers, trials, genes
  concepts/             # Concept pages — mechanisms, treatment paradigms, staging
  avatar/               # Per-user portrait — questions, decisions, preferences
    {user}/             # One subdirectory per user (e.g. jim.chen/)
      questions.md      # Curated, categorized log of queries posed
      decisions.md      # Specific clinical choices with reasoning
      preferences/      # Crystallized practice patterns (one file each)
```

---

## User identity

This wiki supports multiple users. The clinical knowledge layer (`wiki/sources/`, `wiki/entities/`, `wiki/concepts/`) is **shared** across all users — clinical evidence is universal. The avatar layer (`wiki/avatar/{user}/`) is **per-user** — decisions and preferences are personal.

**Default user:** `jim.chen`. All avatar operations target `wiki/avatar/jim.chen/` unless the session explicitly switches users.

**Switching user:** at session start, the user may say *"switch to user X"* (or *"this session is for X"*). The agent then targets `wiki/avatar/X/` for the duration of the session. If `wiki/avatar/X/` does not exist, scaffold it on first avatar write.

**User handles:** lowercase, alphanumeric + dots/hyphens, no spaces. Become directory names under `wiki/avatar/`.

**Scope:** user identity affects only the avatar layer. Ingest, Search, Lint, and shared-knowledge updates are not user-scoped — anything that improves the shared wiki benefits everyone.

## Page types

### Source summaries (`wiki/sources/`)

One page per ingested source. Filename: kebab-case of the source title.

```yaml
---
title: "Full title of the source"
authors: ["Last, First", "Last, First"]
year: 2024
source_type: paper | guideline | note | search
journal: "Journal name"           # if applicable
tags: [cancer/lung, therapy/immunotherapy]
date_ingested: 2026-04-12
---
```

Body structure:
1. **Summary** — 3-5 sentence overview of the source
2. **Key findings** — bullet list of the most important findings or recommendations
3. **Entities mentioned** — wikilinks to entity pages (create if they don't exist)
4. **Relevance** — how this source connects to existing wiki knowledge
5. **Limitations / caveats** — study limitations, conflicts of interest, evidence quality

### Entity pages (`wiki/entities/`)

One page per drug, cancer type, biomarker, gene, clinical trial, or other named entity. Filename: kebab-case.

```yaml
---
title: "Entity name"
entity_type: drug | cancer | biomarker | gene | trial | other
aliases: ["alternative names"]
tags: [therapy/targeted, cancer/lung]
---
```

Body structure:
1. **Overview** — what this entity is
2. **Key facts** — bullet list of important details, each with a wikilink to the source
3. **Related entities** — wikilinks to other entity pages
4. **Sources** — list of source summary wikilinks that mention this entity

### Concept pages (`wiki/concepts/`)

Pages for treatment paradigms, mechanisms of action, staging systems, resistance mechanisms, etc.

```yaml
---
title: "Concept name"
tags: [mechanism, resistance]
---
```

Body structure:
1. **Explanation** — clear description of the concept
2. **Key points** — bullets with source wikilinks
3. **Related entities** — wikilinks
4. **Related concepts** — wikilinks
5. **Sources** — list of source wikilinks

### Avatar pages (`wiki/avatar/{user}/`)

The user's evolving portrait — the questions they ask, the decisions they make, the preferences that crystallize from repeated decisions. One subdirectory per user.

**Provenance carve-out.** The shared-knowledge provenance rule ("every claim wikilinks to a source") **does not apply to avatar content**. Avatar pages instead link to:
- The questions or decisions that generated them (within the user's own avatar)
- The shared entities/concepts they reference (using normal wikilinks)

Avatar content may include user judgment that isn't sourceable to external literature. That is by design.

#### `questions.md`

A curated, categorized log of queries the user has posed. Updated on every non-trivial Query. Each entry:

```markdown
## [YYYY-MM-DD] <short question label>

- **Question:** <verbatim or paraphrased>
- **Trigger:** <case description, source, or context>
- **Wiki pages consulted:** [[entity-1]], [[concept-2]] (or "none — wiki empty for topic")
- **KG tools used:** kg_search, kg_neighbors (or none)
- **Gemini calls:** 0 | 1 | 2+ — <justified | wasteful>
- **Answer origin:** wiki | search | mixed
- **Tokens (Gemini):** <number or 0>
- **Filed back:** [[new-page]] or none
```

#### `decisions.md`

Specific clinical choices the user made, with reasoning. Each entry:

```markdown
## [YYYY-MM-DD] <case label>

- **Scenario:** <brief case>
- **Options considered:** <list>
- **Decision:** <what the user chose>
- **Reasoning:** <user's stated reasoning>
- **Evidence cited:** [[source-1]], [[entity-2]]
- **Confidence:** high | moderate | low
- **Linked question:** [[questions#YYYY-MM-DD-label]]
```

#### `preferences/{kebab-name}.md`

Crystallized practice patterns that emerge from repeated decisions.

```yaml
---
user: jim.chen
title: "Preference title"
tags: [preference]
last_updated: 2026-04-12
crystallized_from: ["[[decisions#YYYY-MM-DD-case-1]]", "[[decisions#YYYY-MM-DD-case-2]]"]
---
```

Body structure:
1. **Preference** — clear statement of the practice pattern
2. **Reasoning** — why this preference exists
3. **Supporting evidence** — wikilinks to shared sources
4. **Caveats / exceptions** — when this preference doesn't apply
5. **Crystallized from** — wikilinks to the decisions that produced this pattern

#### Frontmatter on all avatar pages

Every page in `wiki/avatar/{user}/` carries `user: <handle>` in frontmatter as defense in depth, so a misfiled page is recoverable.

### Overview (`wiki/overview.md`)

A single evolving page that synthesizes the entire wiki at a high level. Updated after every ingest. No rigid structure — write it as a clear narrative that captures the current state of knowledge in the wiki. Link liberally to entity and concept pages.

## Conventions

### Filenames
- All wiki filenames use **kebab-case**: `non-small-cell-lung-cancer.md`, `pembrolizumab.md`
- No spaces, no uppercase in filenames

### Wikilinks
- Use Obsidian-style wikilinks: `[[pembrolizumab]]`, `[[non-small-cell-lung-cancer]]`
- For display text: `[[pembrolizumab|Keytruda]]`
- Links resolve by filename (without extension), so keep filenames unique across all `wiki/` subdirectories

### Provenance rule
**Every factual claim in the wiki must wikilink back to at least one source summary.** This is the most important rule. It prevents hallucination drift — the wiki is only as reliable as its sources. If you cannot link a claim to a source, flag it with `[citation needed]`.

### Tags
Use hierarchical tags in YAML frontmatter. Starter taxonomy:

**Cancer types:**
`#cancer/lung`, `#cancer/breast`, `#cancer/colorectal`, `#cancer/prostate`, `#cancer/hematologic`, `#cancer/pancreatic`, `#cancer/ovarian`, `#cancer/melanoma`, `#cancer/renal`, `#cancer/bladder`, `#cancer/head-and-neck`, `#cancer/gastric`, `#cancer/hepatocellular`, `#cancer/thyroid`, `#cancer/sarcoma`, `#cancer/brain`

**Therapy:**
`#therapy/immunotherapy`, `#therapy/chemotherapy`, `#therapy/targeted`, `#therapy/radiation`, `#therapy/surgery`, `#therapy/combination`, `#therapy/hormonal`, `#therapy/cell-therapy`, `#therapy/ADC`

**Biomarkers:**
`#biomarker/PD-L1`, `#biomarker/MSI`, `#biomarker/TMB`, `#biomarker/HER2`, `#biomarker/EGFR`, `#biomarker/ALK`, `#biomarker/KRAS`, `#biomarker/BRAF`, `#biomarker/ROS1`, `#biomarker/NTRK`, `#biomarker/BRCA`, `#biomarker/MET`

**Guidelines:**
`#guideline/NCCN`, `#guideline/ASCO`, `#guideline/ESMO`, `#guideline/WHO`

**Evidence level:**
`#evidence/RCT`, `#evidence/meta-analysis`, `#evidence/retrospective`, `#evidence/case-report`, `#evidence/phase-1`, `#evidence/phase-2`, `#evidence/phase-3`, `#evidence/real-world`

**Other:**
`#mechanism`, `#resistance`, `#toxicity`, `#staging`, `#screening`, `#supportive-care`, `#palliative`, `#genomics`, `#liquid-biopsy`

New tags may be created during ingestion following these hierarchical patterns (e.g. `#cancer/cholangiocarcinoma`, `#biomarker/RET`).

## Operations

### Ingest

When the user adds a new source to `raw/` and asks you to process it:

1. **Read** the source completely
2. **Discuss** key takeaways with the user — ask what they find most interesting or relevant before writing anything
3. **Create a source summary** page in `wiki/sources/`
4. **Update or create entity pages** — for every drug, cancer type, biomarker, gene, or trial mentioned. If the page exists, add new information with source wikilinks. If it doesn't exist, create it.
5. **Update or create concept pages** — for mechanisms, paradigms, or ideas that deserve their own page
6. **Update or create preference pages** — if the user expresses a preference or practice pattern during discussion
7. **Update `wiki/overview.md`** — incorporate the new source into the high-level synthesis
8. **Update `wiki/index.md`** — add entries for all new pages, update descriptions for modified pages
9. **Append to `wiki/log.md`** — record what was done

A single source may touch 10-15 wiki pages. This is expected.

### Search

When the user wants to retrieve new information from the web:

1. **Formulate a query** — based on what the user wants to know
2. **Run the search script**: `python search.py "query"` (optionally with `--prompt "additional instructions"`)
3. **Read the generated file** — the script prints the file path to stdout
4. **Discuss the results** with the user — highlight key findings, assess source quality
5. **Ingest the file** — follow the standard ingest workflow (steps 3-9 above)

**Provenance note:** Search results are secondary sources — Gemini synthesized them from web content. They have **lower provenance confidence** than primary papers or official guidelines. When ingesting search results:
- Tag claims with `[search-sourced]` if they cannot be traced to a specific primary source
- Prefer upgrading to primary sources when possible (e.g., if a search result cites a specific paper, look for that paper)
- The `sources` list in the search file's frontmatter contains the URLs Gemini used — these are the closest thing to primary provenance

### Query

When the user asks a question:

1. **Read `wiki/index.md`** to find relevant pages
2. **Read the relevant pages** — exhaust the wiki before reaching for external tools
3. **Use graph tools** (`kg_search`, `kg_neighbors`, `kg_paths`) to surface related context the index didn't capture
4. **Decide whether external retrieval is justified** — only call `search.py` (or other web tools) if the wiki has a genuine gap. A query that the wiki can answer should not trigger a Gemini call.
5. **Synthesize an answer** with wikilinks as citations. Distinguish wiki-sourced claims from search-sourced ones (use `[search-sourced]` tag on the latter).
6. **Capture the user's decision** — if the user makes a clinical judgment in response, append it to `wiki/avatar/{user}/decisions.md` per the avatar schema.
7. **Append a query log entry** to `wiki/log.md` AND to `wiki/avatar/{user}/questions.md` with the structured fields below.
8. **Optionally file the answer** — if the synthesis is reusable (comparison, analysis, connection), ask the user whether to save as a new shared page (`wiki/concepts/`) or as an avatar entry.

#### Query log entry format

Every non-trivial Query produces a structured entry in `wiki/log.md`:

```
## [YYYY-MM-DD] query | <short label>

- **User:** <user handle>
- **Question:** <verbatim>
- **Wiki pages consulted:** [[page-1]], [[page-2]] (or "none")
- **KG tools used:** <list> (or "none")
- **Gemini calls:** <count> — <justified | wasteful | none>
- **Answer origin:** wiki | search | mixed
- **Tokens (Gemini):** <total_token_count or 0>
- **Filed back:** [[new-page]] or none
- **Decision captured:** [[avatar/{user}/decisions#YYYY-MM-DD-label]] or none
```

The same entry (or a derived version) appends to `wiki/avatar/{user}/questions.md` so the user's question history is queryable independent of the chronological log.

**Why this matters.** This structure makes test goals 2 and 3 measurable — every query is auditable for whether the wiki was consulted first and whether the Gemini call was justified.

### Supersede

When a new source replaces older information (e.g., a 2026 NCCN update overrides a 2024 recommendation), use **Supersede** rather than the contradiction-flagging convention.

**Rule of thumb:**
- **Peer-level contradiction** (two guidelines disagreeing, two trial readouts conflicting at the same time): flag both, do not overwrite. Contradictions are information.
- **Supersession** (newer source from same/higher authority replaces an older claim): overwrite the wiki content, log the diff in `wiki/log.md`.

Workflow:

1. **Identify** the affected entity/concept page(s)
2. **Rewrite** the affected section with the new claim, citing the new source
3. **Append** a structured entry to `wiki/log.md`:

```
## [YYYY-MM-DD] supersede | <page name>

- **Page:** [[page-name]]
- **Old claim:** <verbatim>
- **Old source:** [[old-source-summary]]
- **New claim:** <verbatim>
- **New source:** [[new-source-summary]]
- **Reason:** <why supersession is appropriate, e.g. "newer guideline same authority"
```

4. Git commit captures the diff for free as a fallback audit trail.

### Lint

Periodically (or when asked), health-check the wiki:

- [ ] **Orphan pages** — pages with no inbound wikilinks
- [ ] **Dead links** — wikilinks that point to nonexistent pages
- [ ] **Missing cross-references** — entities mentioned in text but not wikilinked
- [ ] **Citation needed** — claims without source wikilinks
- [ ] **Stale claims** — information that newer sources may have superseded
- [ ] **Contradictions** — conflicting information across pages
- [ ] **Missing pages** — important entities or concepts mentioned but lacking their own page
- [ ] **Index drift** — pages that exist but aren't in the index

Report findings and fix them (or propose fixes for the user to approve).

## Graph tools (knowledge-graph MCP)

The wiki has a knowledge graph MCP server that parses wikilinks into a graph and provides graph algorithms for retrieval. These tools are available as MCP calls during any Claude Code session.

### Available tools

| Tool | Purpose | When to use |
|------|---------|-------------|
| `kg_search` | Semantic + full-text search across the graph | Finding relevant pages when the index isn't enough |
| `kg_node` | Look up a specific node's metadata and connections | Inspecting an entity and everything linked to it |
| `kg_neighbors` | Explore N-hop connected nodes | "What's connected to KRAS within 2 hops?" |
| `kg_paths` | Find connection routes between two nodes | Tracing how two entities are related (e.g. drug → biomarker → cancer) |
| `kg_common` | Identify shared neighbors between nodes | Finding what two entities have in common |
| `kg_subgraph` | Extract a localized connected component | Getting a focused view of a topic cluster |
| `kg_bridges` | Find bridge nodes (high betweenness centrality) | Identifying key connector pages in the wiki |
| `kg_communities` | List detected topic clusters (Louvain) | Understanding the wiki's thematic structure |
| `kg_community` | Get details of a specific community | Drilling into a topic cluster |
| `kg_central` | Rank nodes by PageRank | Finding the most important/connected pages |
| `kg_index` | Rebuild the graph from current vault state | **Run after every ingest** to keep the graph current |
| `kg_create_node` | Create a new markdown file with frontmatter | Creating new wiki pages via graph tools |
| `kg_annotate_node` | Append content to an existing node | Adding information to existing pages |
| `kg_add_link` | Create a wikilink between two nodes | Adding cross-references |

### When to use graph tools vs. index

- **Use the index** (`wiki/index.md`) for straightforward lookups — "what sources do we have on lung cancer?"
- **Use graph tools** for relational queries — "how is pembrolizumab connected to resistance mechanisms?", "what are the hub pages in the wiki?", "which pages are isolated?"
- **Use graph tools for lint** — `kg_communities` reveals thematic clusters, `kg_bridges` finds key connector pages, `kg_search` can find orphans

### Keeping the graph current

After every ingest operation, run `kg_index` to rebuild the graph. The indexer is incremental (tracks file timestamps), so rebuilds are fast.

## Important rules

1. **Never modify files in `raw/`.** They are immutable source material.
2. **Always maintain provenance.** Every claim links to a source. No exceptions.
3. **Update the index and log on every operation.** These are the wiki's navigation and history.
4. **Use wikilinks liberally.** The graph of connections is as valuable as the content itself.
5. **When in doubt, create a page.** It's better to have a stub entity page than to leave knowledge unlinked. Stubs can be fleshed out later.
6. **Flag contradictions explicitly.** If a new source contradicts existing wiki content, note the contradiction on both pages. Don't silently overwrite — contradictions are information.
7. **Keep the overview current.** After every ingest, update `overview.md` to reflect the new state of knowledge.
8. **Honor user identity.** Default user is `jim.chen`. If a session begins with "switch to user X", target `wiki/avatar/X/` instead. Never write avatar content to the wrong user's directory.
9. **Wiki-first retrieval.** On every Query, exhaust the wiki (index → relevant pages → graph tools) before calling external tools like `search.py`. Each Gemini call must be justifiable as filling a genuine gap.
10. **Supersede vs. flag.** When a new source merely *updates* an older claim from the same/higher authority, supersede (rewrite + log the diff). Only flag contradictions when sources are peer-level and genuinely disagree.
